# 需求清洗系统 (Requirement Cleaning System)

一套专为 **AI 编程人员** 设计的客户需求清洗与管理系统。解决"客户用 AI 工具表达需求时，AI 替客户脑补技术方案、字段设计、接口定义、可行性承诺"导致的需求污染问题。

## 解决的问题

当你使用 Claude Code、Codex、Cursor 等 AI 编程工具完成甲方定制项目时，客户经常使用豆包、DeepSeek、ChatGPT 等工具整理需求。这些 AI 工具在完全不了解你现有项目的情况下，会假设技术栈、设计数据库字段、编造接口、承诺可行性，导致：

- 开发出客户并不需要的功能
- 客户 AI 文档里的技术方案和现有项目冲突
- 客户看到 AI 说"很简单"就认为真的简单，质疑你的能力
- 返工频繁，沟通成本高
- 客户确认了 AI 文档但实际并不理解

本系统通过 **需求分级 + 候选确认 + 技术隔离** 的流程，彻底解决以上问题。

## 核心设计理念

> **AI 生成内容不直接作为需求，也不直接删除；先拆解为业务候选项，再由客户确认。技术实现内容只作为参考，不作为客户需求。**

```
客户原始内容
    ↓
AI内容拆解（五分类）
    ↓
明确需求 + 候选需求 + 技术建议参考 + 无效承诺 + 待澄清
    ↓
候选需求选择题化 → 客户确认
    ↓
形成正式需求 → AI编程工具影响分析
    ↓
确认开发方案 → 开发 → 验收
```

## 五分类体系

| 类别 | 含义 | 处理方式 |
|------|------|---------|
| **明确需求** | 客户明确表达的业务目标、问题、效果 | 直接进入需求文档 |
| **候选需求** | AI 扩展出但可能命中客户意图的业务细节 | 转为选择题让客户确认 |
| **技术建议** | 字段、接口、数据库、框架、代码方案等 | 降级为参考，开发阶段评估 |
| **无效承诺** | "很简单""一定能实现""无需改动"等 | 直接剔除 |
| **待澄清** | 客户表达模糊，无法判断真实含义 | 生成非技术化追问 |

## 安装

### 方式一：一键安装

```bash
# Linux/macOS
bash install.sh

# Windows PowerShell
.\install.ps1
```

### 方式二：手动安装

```bash
# 安装 skills
cp skills/*.md ~/.claude/skills/
cp skills/*.md ~/.codex/skills/

# 安装 templates
mkdir -p ~/.claude/templates/req-system/
mkdir -p ~/.codex/templates/req-system/
cp templates/*.md ~/.claude/templates/req-system/
cp templates/*.md ~/.codex/templates/req-system/
```

## 包含的 Skills

| Skill | 用途 | 触发时机 |
|-------|------|---------|
| `/req-clean` | 需求清洗与候选需求识别 | 收到客户任何形式的原始需求时 |
| `/req-confirm` | 客户确认话术生成 | 清洗结果需要发给客户确认时 |
| `/req-analyze` | 开发前置影响分析 | 客户确认后、正式开发前 |
| `/req-accept` | 验收标准生成 | 开发前（明确目标）和开发后（验收测试） |

## 包含的 Templates

| 模板 | 用途 |
|------|------|
| `project-facts.md` | 项目事实库：仅记录代码中真实存在的事实，不许推测 |
| `candidate-backlog.md` | 候选需求池：暂存未确认的候选需求 |
| `req-doc.md` | 单个需求文档模板：从原始需求到验收的完整记录 |

## 使用流程

### 第 0 步：项目初始化（仅首次）

让 AI 编程工具扫描项目后基于 `project-facts.md` 模板生成项目事实库：

```
请扫描当前项目代码和已有文档，基于 templates/req-system/project-facts.md 模板生成项目事实库。只记录真实存在的事实，不许推测。保存为 /docs/project-facts.md。
```

### 第 1 步：需求清洗

每次收到客户需求时，使用 `/req-clean`：

```
/req-clean

[粘贴客户原始内容]
```

输出包含：明确需求、候选需求、待澄清问题、技术建议参考、无效承诺、给客户确认的话术。

### 第 2 步：客户确认

可直接用 `req-clean` 输出中的"给客户确认的话术"，或用 `/req-confirm` 进一步优化：

```
/req-confirm

[粘贴 req-clean 的输出]
```

关键技巧：把候选需求做成 **选择题** 而不是问答题。客户回复"A、C、F"即可，降低确认门槛。

### 第 3 步：开发前置影响分析

客户确认后，使用 `/req-analyze`：

```
/req-analyze

已确认需求：[粘贴客户确认的需求]
已确认候选项：[粘贴客户选择的候选项]
未确认候选项：[粘贴客户未选的候选项]
```

此阶段 **只分析不开发**。AI 编程工具会结合项目代码全面扫描影响范围，输出开发任务拆分和风险提示。

### 第 4 步：开发

确认影响分析后，进入正式开发。

### 第 5 步：验收

开发完成后，使用 `/req-accept`：

```
/req-accept

已确认需求：[粘贴]
```

生成客户可理解、可实测的验收标准。

## 关键技巧

### 1. 技术表达反推业务意图

AI 写的技术内容不是直接删除，而是反推出业务问题：

| AI 技术内容 | 转化为候选需求 |
|-------------|--------------|
| 新增 status 字段 | 是否需要区分不同状态？状态有哪些？ |
| 新增导出接口 | 是否需要导出数据？导出哪些内容？ |
| 使用管理员权限控制 | 是否只有特定人员能操作？ |
| 新增订单详情页 | 是否需要查看订单完整详情？ |
| 增加通知表 | 是否需要消息通知？通知谁？ |

### 2. 追问必须是非技术语言

| 不要问（技术语言） | 应该问（业务语言） |
|-------------------|-------------------|
| 是否需要 RBAC 权限模型？ | 是所有员工都能用，还是只有特定人员？ |
| 是否采用软删除？ | 删除后是彻底消失，还是以后能找回？ |
| 该字段是否唯一？ | 同一个编号，系统里允许重复吗？ |
| 是否需要状态机？ | 这件事有几个固定阶段？比如待处理→处理中→已完成 |

### 3. 和客户提前建立规则

建议在合作初期给客户发一段说明：

> 为了避免 AI 整理的内容和真实开发情况不一致，后续新增或修改需求时，AI 文档中的技术方案、字段、接口、代码建议和可行性判断不会直接作为开发依据。我们会先提取你真正想要的业务效果，整理成确认版给你确认后再开发。

## 适用场景

- 甲方客户使用 AI 工具整理需求，内容混杂技术假设
- 项目开发过程中客户频繁临时增加需求
- 非技术背景的客户需要确认需求但无法回答技术问题
- 客户AI承诺的内容与实际开发能力不符，导致信任危机
- 需要建立可追溯的需求确认记录
